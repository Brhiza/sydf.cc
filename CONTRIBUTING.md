# 贡献指南

感谢你愿意参与维护时月东方。

这份指南的目标很简单：让每一次改动都更容易 review、更容易回归、更不容易把规则改散。

## 开始之前

请先阅读以下文档：

- [README](./README.md)
- [开发说明](./docs/development.md)
- [架构说明](./docs/architecture.md)
- [开发者 API](./docs/api.md)

## 基本原则

- 所有文档、注释、提交说明与对外可见文本统一使用简体中文
- 行为统一时不要只改前端表现，要同时检查服务端、类型定义与文档
- 优先删除过时分支，而不是继续叠加兼容逻辑
- 修复 bug 时，优先补能复现问题的测试
- 不要把密钥、Token、生产地址配置直接写进公开代码

## 本地开发流程

### 安装依赖

```bash
npm install
```

### 启动开发

```bash
npm run dev
```

### 运行测试

```bash
npm test
```

### 构建验证

```bash
npm run build
```

## 提交前检查

请至少完成以下检查：

1. 受影响的功能在本地手动跑通过
2. `npm run check` 通过
3. 如只需局部验证，至少运行 `npm test` 与 `npm run build`
4. 如果改了接口、页面文案、SEO、构建模式或历史记录逻辑，同步更新文档

## 建议的改动范围

### 适合单独提交的改动

- 单个 bug 修复
- 单个页面交互优化
- 单个服务或工具函数重构
- 单份文档补充或修正

### 不建议混在一个提交里的改动

- 样式大改 + 服务端协议改动
- 占卜规则调整 + 文档重写 + 杂项格式化
- 修复 bug 的同时顺手大规模重命名

## 文档同步规则

出现以下情况时，必须同步更新文档：

- 新增或删除页面入口
- 改动 API 请求字段、响应结构或鉴权方式
- 改动环境变量
- 改动构建命令、部署方式或预渲染流程
- 改动“今日运势”和其他占卜的统一规则

## Issue 建议写法

请尽量提供：

- 问题发生页面
- 复现步骤
- 实际结果
- 预期结果
- 浏览器或设备信息
- 是否和历史记录、追问、重新生成、缓存更新有关

## Pull Request 建议写法

建议包含以下内容：

- 改了什么
- 为什么要改
- 影响范围
- 测试方式
- 是否需要同步部署配置或环境变量

## 涉及高风险区域时请格外小心

以下目录经常牵一发而动全身：

- `src/services/`
- `src/composables/`
- `src/seo/`
- `functions/`
- `src/sw.ts`

如果你改了这些区域，建议额外检查：

- 历史记录是否还能正常回看
- 追问链路是否还能正常续接
- 错误态是否还能统一落到助手消息
- 生产构建与预渲染是否保持一致

## 当前缺失但欢迎补充

- 自动化 CI
- 更明确的部署文档
- 更细的回归测试矩阵
- 明确的开源许可证
